# jianwu v0.3 审计决策（2026-06-30）

> v0.3.5 ship 后整体审计发现 7 项实施层与声明层的偏差。本文归档其中 2 项需 ADR 的（难撤销 + 费解 + 真权衡）。
> 其余 5 项（发布流程、token 可见性、测试缺口、文档同步、waitlist 安全）属于实施层，写在对应 commit message 里足够。
> 完整推理在原始审计会话 transcript；本文是结论速查。

---

## Q1 — v0.3 "SaaS-ready" 重新定义为 single-tenant；v0.4 多租户推迟到触发式启动

**背景：** v0.3.0–v0.3.5 ROADMAP 声称"已全部交付可被多租户 Web 服务安全嵌入的内核"。审计发现：

- `internal/config/secrets.go:54` `var secretsProvider SecretsProvider` + 全局 `SetSecretsProvider(...)` setter——模式与 v0.3.4 刚清掉的 `chatterProviderHook` 同构
- `internal/book/io.go:13` `var DefaultStorage storage.Storage = storage.OS`——包级可变默认
- `internal/cli/root.go:20` `var cliWorkspaceDir string`——`--dir` 进程级共享
- `defaultSecretsProvider.LoadSecretsFor(_ string)` 忽略 tenantID 参数（stub）

v0.3.4"无全局可变 provider 状态"的声明仅覆盖 chatter 装配，未覆盖 secrets/storage/workspace。

**决定：** 把 v0.3 范围**重定义**为 "single-tenant SaaS-ready"——启动期注入 Secrets/Storage/工作目录安全，运行期不可变；不再声称多租户。新增 **v0.4 切片**处理多租户接线（3 全局 → 显式参数 DI、`LoadSecretsFor` 真用 tenantID）。

**v0.4 启动条件改为触发式：** mouqin 上线后由**真实多租户需求触发**（用户数据混杂事故 / 分账需求 / 单租户运维成本可见），不预设范围、不进 ROADMAP 关键路径。

**Why（权衡）：**
- A（单租户先上，v0.4 触发式）→ mouqin v1.0 提前 1-2 周验证产品/市场契合
- B（多租户 day-1，v0.4 在关键路径）→ 架构干净但产品未验证时投资多租户是过度工程

选 A 的关键依据：`internal/storage/namespace.go` 已实现 `Namespace` 前缀隔离（不是文档说的"预留"），让 v0.4 实际工作量从 1-2 周降到 2-3 天，等真实需求再做的成本可承受。

**Consequences：**
- PROJECT_STATUS / ROADMAP 改写：v0.3 表述为 "single-tenant SaaS-ready"，v0.4 列为"触发式启动，不在关键路径"
- mouqin MVP 用 `Storage.Namespace("global")` 单命名空间跑，启动期一次性 `SetSecretsProvider(...)`
- 未来若 v0.4 触发，需同时把"触发事件 + 决策"落到本文件追加章节

**Surprise 候选：** 未来读 ROADMAP 看到"v0.3 已交付 SaaS-ready"+ "mouqin 单租户运行"会困惑——本 ADR 是答案。

## Q2 — mouqin.com 官网留在 jianwu 仓库（co-hosted）

**背景：** 三份文档（`PROJECT_STATUS.md §2` / `ROADMAP.md §v1.0` / `CONTEXT.md` 隐含）一致说 "mouqin = 独立仓库，v1.0 未启动"。但仓库实际包含：

```
scripts/deploy-mouqin.sh           # Cloudflare Pages + Resend 完整部署脚本
website/                           # 完整 Hugo 站
  content/{docs,blog,engine,developers}/
  functions/api/waitlist.js        # 生产 waitlist API
  hugo.toml, layouts/, assets/
docs/DEPLOY_MOUQIN.md              # 部署手册
```

commit `07cd538` 已标注 "v0.3.5 — 官网上线 + mouqin 部署脚本"——mouqin.com **已经在跑**，不是"v1.0 未启动"。

**决定：** mouqin.com 官网**留在 jianwu 仓库**。"独立仓库 mouqin" 语义收窄为"未来 SaaS app 仓库"——官网不是 SaaS 本体，是 jianwu 项目的营销/文档面。

**接受的代价（已知情接受）：**
1. `go install github.com/iannil/jianwu/cmd/jianwu@latest` 下载完整 Hugo 资源（无关体积）
2. license 边界：jianwu AGPL-3.0 vs 官网内容 ©祝融 / All Rights Reserved——用文件级 copyright 注释解，必要时加 `website/LICENSE`
3. commit 历史耦合：jianwu 引擎迭代与 mouqin.com GA 调整在同一 log

**Why（权衡）：**
- A（拆 mouqin-site 独立仓库）→ 干净但 commit `07cd538` 已绑定"官网上线"与"v0.3.5 ship"，越拖越难拆；当前官网内容少，迁移成本低于维护分离的成本
- B（留）→ 接受上述 3 个代价；未来若 SaaS 启动且需要独立品牌资产再评估

选 B 的关键依据：mouqin.com 内容量小（一个 Hugo 站 + 一个 waitlist 函数），3 个代价目前都低于"拆仓 + 重新接 CI + 双仓库同步"的成本。

**Consequences：**
- PROJECT_STATUS §2 改为三角色：`jianwu`（引擎+CLI）/ `mouqin.com` 官网（co-hosted in jianwu repo）/ `mouqin` SaaS app（未来独立仓库，v1.0 未启动）
- 若未来拆仓，迁移项清单：`website/` + `scripts/deploy-mouqin.sh` + `docs/DEPLOY_MOUQIN.md` + Cloudflare Pages 项目指向新仓库
- 不阻塞 v0.3.x / v0.4 任何切片

**Surprise 候选：** 未来读 `PROJECT_STATUS` 看到 mouqin 表述变化会困惑——本 ADR 是答案。
